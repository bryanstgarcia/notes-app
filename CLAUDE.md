## What this project is

A full-stack web application with a Django REST API backend and a Next.js frontend, authenticated via JWT. It's a notes app: users register/log in and manage categorized notes (create, list, view, inline-edit with autosave). Two Django apps exist on the backend (`users`, `notes`), and the frontend has real feature modules for auth, the notes dashboard, and note detail/editing. `frontend/src/types/` remains an empty placeholder — domain types come from the generated API client.

## Architecture

Monorepo with two independent projects that communicate **exclusively via HTTP** — neither imports code from the other:
- `backend/`: Django REST API. Exposes endpoints under `/api/`.
- `frontend/`: Next.js (App Router) client that consumes that API.

The contract between them is an OpenAPI schema generated by `drf-spectacular` from the DRF views/serializers, saved to `backend/openapi.json`/`frontend/openapi.json`, and used to generate a typed client in `frontend/src/lib/api/` via `openapi-typescript-codegen`.

In dev, the frontend proxies `/api/*` requests to the backend container via a Next.js rewrite (`frontend/next.config.ts`) pointing at `http://backend:8000/api/:path*/` — there is no Nginx reverse proxy. CORS is also enabled via `django-cors-headers` (`CORS_ALLOWED_ORIGINS` includes `http://localhost:3000`).

### Auth flow

JWT auth via `djangorestframework-simplejwt`, with `AUTH_USER_MODEL = "users.User"` (an `AbstractUser` subclass in `backend/apps/users/models.py` with a unique, required `email` used as `USERNAME_FIELD`). Endpoints under `/api/auth/` (`backend/apps/users/urls.py`):
- `POST /api/auth/register/` — create a user by email + password (`AllowAny`). The username is always server-generated from the email local-part (see `apps/users/services.py`) — client-supplied usernames are ignored.
- `POST /api/auth/login/` — obtain access + refresh tokens, logging in by **email** (`TokenObtainPairView`)
- `POST /api/auth/refresh/` — refresh an access token (`ROTATE_REFRESH_TOKENS = True`)
- `GET /api/auth/me/` — retrieve the authenticated user

Known limitation: login email lookup is case-sensitive (documented in a comment in `models.py`), so a user who logs in with different casing than they registered with will get a spurious 401.

DRF defaults (`backend/config/settings.py`) require JWT auth and `IsAuthenticated` on every view unless overridden — new endpoints are locked down by default.

### Notes feature

`apps.notes` (`backend/apps/notes/`) exposes notes scoped to their owner, mounted at `/api/notes/`:
- `GET /api/notes/` — list the current user's notes, newest first
- `POST /api/notes/` — create a note owned by the current user (client-supplied `owner`/`owner_id` is ignored)
- `GET /api/notes/<id>/` — retrieve a single note; returns 404 (not 403) if it exists but belongs to another user, to avoid leaking existence
- `PATCH /api/notes/<id>/` — partially update a note; `PUT` and `DELETE` are explicitly disabled (405)

Each `Note` has a `category` (`random-thoughts` | `school` | `personal`, default `random-thoughts`), `title`, and `content`. Business logic (ownership checks, scoped queries) lives in `apps/notes/services.py`, not in the views.

On the frontend, the notes dashboard (`(protected)/dashboard`) lists notes with category filtering, and `(protected)/note/[id]` is the detail/edit view with debounced content autosave (`AUTOSAVE_DEBOUNCE_MS = 800`), save-on-blur for the title, and immediate save for category changes.

## How to start the project

`make dev` from the root. Requires Docker. Copy `.env.example` to `.env` first.

- Frontend: `http://localhost:3000`
- Backend API: `http://localhost:8000/api/`
- Swagger UI: `http://localhost:8000/api/docs/`

## Common commands

All run from the repo root via the Makefile (wraps `docker compose`):

| Command | Description |
|---|---|
| `make dev` | Start backend and frontend |
| `make build` | Build both Docker images (run after changing `requirements.txt`/`package.json`) |
| `make down` | Stop and remove containers |
| `make logs` | Follow logs from all services |
| `make migrate` | Apply Django database migrations |
| `make makemigrations` | Generate new Django migrations |
| `make createsuperuser` | Create a Django admin superuser |
| `make shell` | Open a Django shell |
| `make reset-db` | Wipe the SQLite volume and re-run migrations |
| `make generate-schema` | Generate `backend/openapi.json` and copy it to `frontend/openapi.json` |

Backend tests use Django's test runner, e.g.:
```bash
docker compose run --rm backend python manage.py test apps.users
docker compose run --rm backend python manage.py test apps.notes
docker compose run --rm backend python manage.py test apps.users.tests.SomeTestCase.test_something  # single test
```

Frontend lint: `npm run lint` inside `frontend/` (or `docker compose run --rm frontend npm run lint`). No frontend test runner is configured yet.

## Backend conventions

- Django apps live in `backend/apps/<name>/`.
- Each app has: `models.py`, `serializers.py`, `views.py`, `urls.py`, `tests.py`, `services.py`, `constants.py`.
- Business logic (ownership checks, scoped lookups, username generation, etc.) belongs in `services.py`, not in views or serializers.
- Magic numbers/strings (max lengths, retry counts, error messages) belong in `constants.py`.
- New endpoints are registered in `backend/config/urls.py`.
- After changing models: `make makemigrations` then `make migrate`.
- After changing endpoints: `make generate-schema` to update the frontend client.

## Frontend conventions

- Uses Next.js App Router. Routes go in `frontend/src/app/`, grouped into `(auth)` (login/register, no navbar) and `(protected)` (requires an active session) route groups.
- The API client is in `frontend/src/lib/api/` and is **auto-generated** via `npm run generate-api` (requires `frontend/openapi.json` to be current). Do not edit manually.
- For backend calls, import from `src/lib/api/` — never use `fetch` directly.
- Feature code lives in `frontend/src/features/<feature>/`, split into `hooks/` (state/logic) and `components/` (presentational). Route-level container components (e.g. `LoginView`, `NotesDashboardView`, `NoteDetailView`) compose these.
- Shared, reusable UI primitives (`Button`, `Input`, `TextLink`, `CloseButton`, `ColorDropdown/`, `NoteCard/`) live in `frontend/src/components/ui/`.
- Auth/session state (`AuthProvider`/`useAuth`, token storage) lives in `frontend/src/features/auth/store/`, not in `lib/`.
- General-purpose, non-generated utilities live in `frontend/src/utils/` (separate from `lib/api/`).
- Styles with Tailwind CSS. No CSS modules or styled-components.
- Strict TypeScript (`strict: true`). No `any` unless justified with a comment.
- Path alias `@/*` maps to `frontend/src/*`.

## OpenAPI schema workflow

Whenever the backend API changes:
1. `make generate-schema` from the root — generates `backend/openapi.json` and copies it to `frontend/openapi.json`.
2. `npm run generate-api` inside `frontend/` — regenerates the typed client in `frontend/src/lib/api/`.

Both steps are required; the generated client is only as current as the last time both were run.

## Database

SQLite in development. The file lives in the Docker volume `sqlite_data` (mounted at `/app/db/db.sqlite3` in the backend container, path controlled by `DB_PATH`).
To reset: `make reset-db`.

## Environment variables

See `.env.example`. Copy to `.env` before starting.
Sensitive variables never go in code — always in `.env`.

## What NOT to do

- Do not install dependencies directly inside the container. Add them to `requirements.txt` or `package.json` and run `make build`.
- Do not edit files in `frontend/src/lib/api/` — they are auto-generated.
- Do not add business logic to Django views; that belongs in serializers or a services module.
